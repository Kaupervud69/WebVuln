**Внедрение Unicode (Unicode Injection)**

**Введение**

В зависимости от того, как серверная или клиентская часть ведет себя при получении странных символов юникода, злоумышленник может обойти защиту и внедрить произвольные символы, которые можно использовать для эксплуатации таких уязвимостей, как XSS или SQL-инъекции.

**Нормализация Unicode (Unicode Normalization)**

Нормализация Unicode происходит, когда символы юникода нормализуются до ASCII-символов.

Один из распространенных сценариев такого типа уязвимости возникает, когда система каким-либо образом изменяет ввод пользователя после его проверки. Например, в некоторых языках простой вызов для преобразования ввода в верхний или нижний регистр может нормализовать заданный ввод, и юникод будет преобразован в ASCII, генерируя новые символы.

**\u в %**

Символы Unicode обычно представляются с префиксом `\u`. Например, символ `㱋` — это `\u3c4b` (проверьте здесь). Если серверная часть преобразует префикс `\u` в `%`, результирующая строка будет `%3c4b`, которая в URL-декодировании равна `<4b`. И, как вы можете видеть, внедряется символ `<`.

Вы можете использовать эту технику для внедрения любого типа символов, если серверная часть уязвима.

Проверьте https://unicode-explorer.com/ для поиска нужных символов.

Эта уязвимость на самом деле происходит из уязвимости, найденной исследователем. Для более глубокого объяснения проверьте https://www.youtube.com/watch?v=aUsAHb0E7Cg

**Внедрение эмодзи (Emoji Injection)**

Серверная часть иногда ведет себя странно, когда получает эмодзи. Именно это произошло в этом решении, где исследователь смог добиться XSS с такой полезной нагрузкой: `💋img src=x onerror=alert(document.domain)//💛`

В этом случае ошибка заключалась в том, что сервер после удаления вредоносных символов преобразовал UTF-8 строку из Windows-1252 в UTF-8 (по сути, кодировка ввода и преобразование из кодировки не совпадали). Затем это не дало правильный символ `<`, а только странный символ юникода: `‹`. Затем они взяли этот вывод и снова преобразовали его из UTF-8 в ASCII. Это нормализовало `‹` в `<`. Вот так эксплойт мог сработать в этой системе.

Вот что произошло:

```php
<?php

$str = isset($_GET["str"]) ? htmlspecialchars($_GET["str"]) : "";

$str = iconv("Windows-1252", "UTF-8", $str);
$str = iconv("UTF-8", "ASCII//TRANSLIT", $str);

echo "String: " . $str;
```

**Windows Best-Fit/Worst-fit**

Как объясняется в этой замечательной статье, в Windows есть функция под названием Best-Fit, которая заменяет символы юникода, которые не могут быть отображены в режиме ASCII, на похожие. Это может привести к неожиданному поведению, когда серверная часть ожидает определенный символ, но получает другой.

Возможно найти символы best-fit на https://worst.fit/mapping/.

Поскольку Windows обычно преобразует строки юникода в строки ASCII на одном из последних этапов выполнения (обычно переходя от API с суффиксом "W" к API с суффиксом "A", например, `GetEnvironmentVariableA` и `GetEnvironmentVariableW`), это позволит атакующим обходить защиту, отправляя символы юникода, которые в конечном итоге будут преобразованы в ASCII-символы, выполняющие неожиданные действия.

В сообщении блога предлагаются методы обхода уязвимостей, исправленных с использованием черного списка символов, эксплуатации обхода пути с использованием символов, отображаемых на `/` (0x2F) и символов, отображаемых на `\` (0x5C), или даже обхода защит экранирования оболочки, таких как `escapeshellarg` в PHP или `subprocess.run` в Python со списком. Это было сделано, например, с использованием двойных кавычек полной ширины (U+FF02) вместо двойных кавычек, так что в итоге то, что выглядело как 1 аргумент, было преобразовано в 2 аргумента.

Обратите внимание, что для того, чтобы приложение было уязвимо, оно должно использовать Windows API "W", но в конечном итоге вызывать Windows API "A", чтобы была создана "Best-fit" строка юникода.

Некоторые обнаруженные уязвимости не будут исправлены, так как люди не могут договориться о том, кто должен исправлять эту проблему.
