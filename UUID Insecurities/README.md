**Небезопасность UUID (UUID Insecurities)**

**Основная информация**

Универсальные уникальные идентификаторы (UUID) — это 128-битные числа, используемые для уникальной идентификации информации в компьютерных системах. UUID необходимы в приложениях, где требуются уникальные идентификаторы без централизованной координации. Они обычно используются в качестве ключей базы данных и могут ссылаться на различные элементы, такие как документы и сессии.

UUID разработаны так, чтобы быть уникальными и трудными для угадывания. Они структурированы в определенном формате, разделены на пять групп и представлены 32 шестнадцатеричными цифрами. Существуют различные версии UUID, каждая из которых служит разным целям:

*   **UUID v1** основан на времени, включает временную метку, тактовую последовательность и идентификатор узла (MAC-адрес), но может потенциально раскрывать информацию о системе.
*   **UUID v2** похож на v1, но включает модификации для локальных доменов (широко не используется).
*   **UUID v3 и v5** генерируют UUID с использованием хеш-значений из пространства имен и имени, при этом v3 использует MD5, а v5 — SHA-1.
*   **UUID v4** генерируется почти полностью случайным образом, обеспечивая высокий уровень анонимности, но с незначительным риском дублирования.

**Подсказка**

Обратите внимание, что версия и подверсия UUID обычно находятся в одной и той же позиции внутри UUID. Например, в:
`12345678 - abcd - 1a56 - a539 - 103755193864`
`xxxxxxxx - xxxx - Mxxx - Nxxx - xxxxxxxxxxxx`

*   Позиция `M` указывает версию UUID. В приведенном выше примере это UUID v1.
*   Позиция `N` указывает вариант UUID.

**"Сэндвич-атака" (Sandwich Attack)**

"Сэндвич-атака" — это особый тип атаки, который использует предсказуемость генерации UUID v1 в веб-приложениях, особенно в таких функциях, как сброс пароля. UUID v1 генерируется на основе времени, тактовой последовательности и MAC-адреса узла, что может сделать его в некоторой степени предсказуемым, если злоумышленник может получить некоторые из этих UUID, сгенерированных близко по времени.

**Пример**

Представьте веб-приложение, которое использует UUID v1 для генерации ссылок для сброса пароля. Вот как злоумышленник может использовать это для получения несанкционированного доступа:

1.  **Начальная настройка:**
    *   У злоумышленника есть контроль над двумя учетными записями электронной почты: `attacker1@acme.com` и `attacker2@acme.com`.
    *   Целевая учетная запись жертвы — `victim@acme.com`.

2.  **Выполнение:**
    *   Злоумышленник инициирует сброс пароля для своей первой учетной записи (`attacker1@acme.com`) и получает ссылку для сброса пароля с UUID, например, `99874128-7592-11e9-8201-bb2f15014a14`.
    *   Сразу после этого злоумышленник инициирует сброс пароля для учетной записи жертвы (`victim@acme.com`), а затем быстро для второй подконтрольной учетной записи (`attacker2@acme.com`).
    *   Злоумышленник получает ссылку для сброса для второй учетной записи с UUID, например, `998796b4-7592-11e9-8201-bb2f15014a14`.

3.  **Анализ:**
    *   Теперь у злоумышленника есть два UUID, сгенерированных близко по времени (`99874128` и `998796b4`). Учитывая последовательный характер основанных на времени UUID, UUID для учетной записи жертвы, скорее всего, будет находиться между этими двумя значениями.

4.  **Атака перебором (Brute Force Attack):**
    *   Злоумышленник использует инструмент для генерации UUID между этими двумя значениями и проверяет каждый сгенерированный UUID, пытаясь получить доступ к ссылке сброса пароля (например, `https://www.acme.com/reset/<UUID>`).
    *   Если веб-приложение не ограничивает должным образом скорость запросов или не блокирует такие попытки, злоумышленник может быстро проверить все возможные UUID в этом диапазоне.

5.  **Получение доступа:**
    *   Как только будет обнаружен правильный UUID для ссылки сброса пароля жертвы, злоумышленник может сбросить пароль жертвы и получить несанкционированный доступ к ее учетной записи.
