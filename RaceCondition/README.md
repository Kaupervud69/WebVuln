# Состояние гонки (Race Condition)

> Состояние гонки может возникнуть, когда процесс критически или неожиданно зависит от последовательности или времени выполнения других событий. В среде веб-приложений, где одновременно может обрабатываться несколько запросов, разработчики могут полагаться на то, что конкурентность будет обработана фреймворком, сервером или языком программирования.

# Методология

1. Прогнозирование
* Является ли безопасность этой конечной точки критической?
* Есть ли потенциальная возможность столкновения?
2. Исследование 
* протестировать поведение конечной точки в обычных условиях
* отправить одну и ту же группу запросов одновременно
* найти какую-то форму отклонения
	* изменение одного или нескольких ответов
	* разное содержимое электронной почты
 	* видимое изменение поведения приложения после этого.
3. Доказательство
* Понять, что происходит, удалить лишние запросы и убедиться, все еще можно воспроизвести эффекты.

## Превышение лимита (Limit-overrun)

Превышение лимита относится к сценарию, когда несколько потоков или процессов конкурируют за обновление или доступ к общему ресурсу, что приводит к превышению ресурсом его предполагаемых лимитов.

* **Основная проблема** — синхронизация запросов таким образом, чтобы по крайней мере два окна гонки выстроились в линию, вызывая столкновение. Это окно часто составляет всего миллисекунды и может быть даже короче.

**Примеры:** Превышение лимита расходования средств, множественное голосование, многократное использование подарочной карты.

*   [Состояние гонки позволяет многократно активировать подарочные карты, что приводит к получению бесплатных "денег" - @muon4](https://hackerone.com/reports/759247)
*   [Состояния гонки можно использовать для обхода лимита приглашений - @franjkovic](https://hackerone.com/reports/115007)
*   [Регистрация нескольких пользователей по одному приглашению - @franjkovic](https://hackerone.com/reports/148609)

## Обход ограничения частоты запросов (Rate-limit Bypass)

Обход ограничения частоты запросов происходит, когда пользователь использует отсутствие правильной синхронизации в механизмах ограничения частоты, чтобы превысить допустимые лимиты запросов. Ограничение частоты предназначено для контроля частоты действий (например, API-запросов, попыток входа в систему), но состояния гонки могут позволить злоумышленникам обойти эти ограничения.

**Примеры:** Обход механизмов защиты от перебора паролей и двухфакторной аутентификации (2FA).

*   [Состояние гонки в механизме сброса пароля Instagram - Laxman Muthiyah](https://youtu.be/4O9FjTMlHUM)

## Скрытые многошаговые последовательности

Один запрос может инициировать целую многошаговую последовательность за кулисами, переводя приложение через несколько скрытых состояний «подсостояния», в которые оно входит и из которых снова выходит до завершения обработки запроса. 

* неисправные процессы многофакторной аутентификации (MFA), которые позволяют выполнить первую часть входа с использованием известных учетных данных, а затем перейти прямо к приложению с помощью принудительного просмотра, фактически полностью обходя MFA.

```python
session['userid'] = user.userid
if user.mfa_enabled:
session['enforce_mfa'] = True

* сгенерировать и отправить код MFA пользователю
* перенаправить браузер на форму ввода кода MFA
```

## Race condition нескольких конечных точек

Классический логический изъян в интернет-магазинах, при добавлении товар в корзину и его оплате, сразу добавить в корзину еще больше товаров, прежде чем принудительно перейти на страницу подтверждения заказа.

> Разновидность этой уязвимости может возникнуть, когда проверка платежа и подтверждение заказа выполняются во время обработки одного запроса.
> В этом случае потенциально можно добавить больше товаров в свою корзину в течение окна гонки между моментом проверки платежа и моментом окончательного подтверждения заказа.

* Распространенная проблема в основном вызвана следующими двумя факторами:
> Задержки, вносимые сетевой архитектурой - когда сервер front-end устанавливает новое соединение с back-end может возникнуть задержка. Используемый протокол также может иметь большое влияние.
>
> Задержки, вызванные обработкой, специфичной для конечной точки - Различные конечные точки изначально различаются по времени обработки, иногда значительно, в зависимости от того, какие операции они запускают

*	**Разогрев соединения**
> Если время отклика на одной конечной точке непоследовательное, даже при использовании техники с одним пакетом, это признак того, что задержка на стороне сервера мешает атаке. Это можно обойти используя Turbo Intruder для отправки некоторых запросов на разогрев соединения перед тем, как продолжить с основными запросами на атаку. В Burp Repeater можно попробовать добавить запрос GET для домашней страницы в начало группы вкладок, а затем использовать опцию `Отправить группу последовательно (одно соединение)`-`Send group in sequence (single connection)`.

*	**Злоупотребление ограничениями скорости или ресурсов**
> Веб-серверы часто задерживают обработку запросов, если их слишком много и отправляются слишком быстро. Отправляя большое количество фиктивных запросов, чтобы намеренно вызвать ограничение скорости или ресурсов, можно вызвать подходящую задержку на стороне сервера. Это делает атаку с одним пакетом жизнеспособной, даже когда требуется отложенное выполнение.


## Условия гонки на одной конечной точке

Отправка параллельных запросов с разными значениями на одну конечную точку.

* Механизм сброса пароля, который сохраняет идентификатор пользователя и токен сброса в сеансе пользователя.

Отправка двух параллельных запросов на сброс пароля из одного сеанса, но с двумя разными именами пользователей, может потенциально вызвать следующую коллизию:

```python
session['reset-user'] = victim
session['reset-token'] = hacker
```
Теперь сессия содержит идентификатор пользователя victim, но действительный токен сброса отправляется hacker.

> Чтобы эта атака сработала, различные операции, выполняемые каждым процессом, должны происходить в правильном порядке. Потребуется несколько попыток или немного удачи, чтобы достичь желаемого результата. Электронные письма часто отправляются в фоновом потоке после того, как сервер отправляет HTTP-ответ клиенту, что повышает вероятность условий гонки.

## Механизмы блокировки на основе сеанса

> Если все запросы обрабатываются последовательно, можно попробовать отправить каждый из них с использованием другого токена сеанса.

## Состояния гонки частичного построения

Многие приложения создают объекты в несколько этапов. Это может ввести временное промежуточное состояние, в котором объект может быть использован.

Такое поведение открывает путь для эксплойтов, посредством которых внедряется входное значение, которое возвращает что-то, совпадающее с неинициализированным значением базы данных, например, пустую строку или null в JSON, и это сравнивается как часть контроля безопасности.

* Фреймворки часто позволяют передавать массивы и другие нестроковые структуры данных, используя нестандартный синтаксис. Например, в PHP:

* param[]=foo эквивалентно param = ['foo']
* param[]=foo&param[]=bar эквивалентно param = ['foo', 'bar']
* param[] эквивалентно param = []

Ruby on Rails позволяет делать что-то подобное, предоставляя запрос или параметр POST с ключом, но без значения. 
```
param[key] приводит к следующему объекту на стороне сервера:
params = {"param"=>{"key"=>nil}}
```
Это означает, что во время окна гонки потенциально можно делать аутентифицированные запросы API следующим образом:
```
GET /api/user/info?user=victim&api-key[]= HTTP/2
Хост: victim-website.com
```

## Атаки, чувствительные к времени

Иногда нельзя обнаружить условия гонки, но методы доставки запросов с точным временем все еще могут выявить наличие других уязвимостей.

Одним из таких примеров является использование временных меток высокого разрешения вместо криптографически защищенных случайных строк для генерации токенов безопасности.

* Токен сброса пароля, который рандомизирован только с помощью временной метки
> В этом случае можно запустить два сброса пароля для двух разных пользователей, которые оба используют один и тот же токен. Нужно рассчитать время запросов так, чтобы они генерировали одну и ту же временную метку.

* **Наблюдения**:

* Каждый запрос на сброс генерирует ссылку с разным токеном.
* Токен имеет постоянную длину. Это указывает на то, что это либо случайно сгенерированная строка фиксированной длины, либо хеш каких-то данных, который можно предсказать.
* Тот факт, что токен каждый раз разный, говорит о том, что если это хеш, то в него обязательно входит изменяющееся состояние: например, счётчик, таймер (timestamp) или генератор случайных чисел (RNG).

# Техники

## Синхронизация последнего байта в HTTP/1.1

Отправь все запросы, кроме последнего байта, затем "освободи" каждый запрос, отправив последний байт.

Выполнение синхронизации последнего байта с помощью Turbo Intruder:

```python
engine.queue(request, gate='race1')
engine.queue(request, gate='race1')
engine.openGate('race1')
```

## Атака одним пакетом в HTTP/2

В HTTP/2 вы можете отправлять несколько HTTP-запросов одновременно через одно соединение. При атаке одним пакетом будет отправлено около 20-30 запросов, и они достигнут сервера одновременно. Использование одного запроса устраняет сетевую нестабильность (jitter).

*   [PortSwigger/turbo-intruder/race-single-packet-attack.py](https://github.com/PortSwigger/turbo-intruder/blob/master/resources/examples/race-single-packet-attack.py)

*   **Burp Suite**
	*  Запрос в Repeater.
	*  Продублировать запрос 20 раз (CTRL+R).
	*  Создать новую группу и добавить все запросы.
	*  Отправить группу параллельно (single packet attack).

* `Send group in sequence (single connection)` - Отправить группу последовательно (одно подключение), Repeater установит соединение с целью, отправит запросы со всех вкладок в группе, а затем закроет соединение.
Это полезно для атак на основе синхронизации, которые полагаются на возможность сравнивать ответы с очень небольшими различиями во времени.
* `Send group in sequence (separate connections)` - Отправить группу последовательно (отдельные соединения), Repeater устанавливает соединение с целью, отправляет запрос с первой вкладки, а затем закрывает соединение. Он 							повторяет этот процесс для всех остальных вкладок в том порядке, в котором они расположены в группе.
* `Send group in parallel` -  Repeater синхронизирует параллельные запросы, чтобы гарантировать, что все они прибудут полностью одновременно. Он использует различные методы синхронизации в зависимости от используемой версии HTTP.
*	используя атаку с одним пакетом (или синхронизацию последнего байта, если HTTP/2 не поддерживается)

* **Пример:**

*   [CVE-2022-4037 - Обнаружение состояния гонки в Gitlab с помощью атаки одним пакетом - James Kettle](https://youtu.be/Y0NVIVucQNE)

## Turbo Intruder

* **Пример 1**

1.  Отправьте запрос в Turbo Intruder.
2.  Используйте этот код Python в качестве полезной нагрузки Turbo Intruder:

```python
def queueRequests(target, wordlists):
    engine = RequestEngine(endpoint=target.endpoint,
                        concurrentConnections=30,
                        requestsPerConnection=30,
                        pipeline=False
                        )

    for i in range(30):
        engine.queue(target.req, i)
        engine.queue(target.req, target.baseInput, gate='race1')

    engine.start(timeout=5)
    engine.openGate('race1')
    engine.complete(timeout=60)

def handleResponse(req, interesting):
    table.add(req)
```

3.  Теперь установите внешний HTTP-заголовок `x-request: %s` - ⚠️ Это необходимо для Turbo Intruder.
4.  Нажмите "Attack" (Атака).

* **Пример 2**

Следующий шаблон можно использовать, когда вам нужно отправить запрос2 в состоянии гонки сразу после запроса1, и временное окно может составлять всего несколько миллисекунд.

```python
def queueRequests(target, wordlists):
    engine = RequestEngine(endpoint=target.endpoint,
                           concurrentConnections=30,
                           requestsPerConnection=100,
                           pipeline=False
                           )
    request1 = '''
POST /target-URI-1 HTTP/1.1
Host: <REDACTED>
Cookie: session=<REDACTED>

parameterName=parameterValue
    '''

    request2 = '''
GET /target-URI-2 HTTP/1.1
Host: <REDACTED>
Cookie: session=<REDACTED>
    '''

    engine.queue(request1, gate='race1')
    for i in range(30):
        engine.queue(request2, gate='race1')
    engine.openGate('race1')
    engine.complete(timeout=60)

def handleResponse(req, interesting):
    table.add(req)
```

# ЗаЩитОчка

*    Не допускать смешивания данных из разных хранилищ и баз данных. Это снизит вероятность рассогласованности данных и, как следствие, уязвимостей, связанных с многопоточной обработкой.
*    Использовать механизмы синхронизации, такие как блокировки (mutex), семафоры, условные переменные и т. д. Это предотвратит одновременный доступ к общим ресурсам из разных потоков. А ещё эти механизмы обеспечат корректную последовательность операций при доступе к общим ресурсам, гарантируя, что изменения данных выполняются безопасно и последовательно.
*    Обеспечивать атомарность состояния критичных конечных точек. Здесь помогут функции параллелизма хранилища данных. Например, для проверки соответствия суммы оплаты в корзине и подтверждения заказа необходимо использовать одну транзакцию базы данных. Это предотвратит частичное выполнение операций и гарантирует целостность данных в случае ошибки.
*    Использовать возможности целостности и согласованности хранилища данных. Например, ограничения уникальности столбцов. Эти механизмы защитят от ситуаций, когда несколько потоков могут записать конфликтующие данные, нарушая согласованность хранилища.
*    Не использовать один уровень хранения данных для защиты другого. Например, сеансы не подходят для предотвращения атак на превышение лимитов в базах данных.
*    Проверять систему обработки сеансов на предмет поддержки внутренней согласованности сеансов. Это предотвратит возможные конфликты между сессиями и снизит вероятность атак, связанных с нарушением целостности данных.
*    Избегать глобальных переменных и отдавать предпочтение локальным данным, которые не могут быть изменены несколькими потоками одновременно. Это снизит вероятность возникновения Race Condition и улучшит безопасность приложения.
