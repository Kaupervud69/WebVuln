**Обход правил контроля доступа Nginx с помощью манипуляции путем (Pathname Manipulation)**

Техники из этого исследования.

**Пример правила Nginx:**

```nginx
location = /admin {
    deny all;
}

location = /admin/ {
    deny all;
}
```

Чтобы предотвратить обходы, Nginx выполняет нормализацию пути перед его проверкой. Однако, если внутренний сервер выполняет другую нормализацию (удаляет символы, которые не удаляет nginx), это может позволить обойти эту защиту.

**PHP-FPM**

Конфигурация Nginx для FPM:

```nginx
location = /admin.php {
    deny all;
}

location ~ \.php$ {
    include snippets/fastcgi-php.conf;
    fastcgi_pass unix:/run/php/php8.1-fpm.sock;
```

Nginx настроен на блокировку доступа к `/admin.php`, но это можно обойти, обратившись по адресу `/admin.php/index.php`.

**Как предотвратить:**

```nginx
location ~* ^/admin {
    deny all;
}
```

**Обход правил Mod Security**

**Путаница с путем (Path Confusion)**

В этом посте объясняется, что ModSecurity v3 (до версии 3.0.12) неправильно реализовывал переменную `REQUEST_FILENAME`, которая должна была содержать путь доступа (до начала параметров). Это происходило из-за выполнения декодирования URL для получения пути.

Следовательно, запрос вида `http://example.com/foo%3f';alert(1);foo=` в Mod Security будет предполагать, что путь — это просто `/foo`, так как `%3f` преобразуется в `?`, завершающий путь URL. Но на самом деле путь, который получит сервер, будет `/foo%3f';alert(1);foo=`.

Переменные `REQUEST_BASENAME` и `PATH_INFO` также были подвержены этой ошибке.

Нечто подобное произошло в версии 2 Mod Security, что позволяло обойти защиту, предотвращающую доступ пользователей к файлам с определенными расширениями, связанными с резервными копиями (например, `.bak`), просто отправив точку в URL-кодировке `%2e`. Например: `https://example.com/backup%2ebak`.

**Обход ACL AWS WAF**

**Неправильно сформированный заголовок (Malformed Header)**

В этом исследовании упоминается, что можно было обойти правила AWS WAF, применяемые к HTTP-заголовкам, отправив "неправильно сформированный" заголовок, который не был правильно проанализирован AWS, но был проанализирован внутренним сервером.

Например, отправка следующего запроса с SQL-инъекцией в заголовке `X-Query`:

```
GET / HTTP/1.1\r\n
Host: target.com\r\n
X-Query: Value\r\n
\t' or '1'='1' -- \r\n
Connection: close\r\n
\r\n
```

Позволяла обойти AWS WAF, потому что он не понимал, что следующая строка является частью значения заголовка, в то время как сервер NODEJS понимал (это было исправлено).

**Общие методы обхода WAF**

**Ограничения размера запроса (Request Size Limits)**

Обычно WAF имеют определенный лимит длины запросов для проверки, и если POST/PUT/PATCH-запрос превышает его, WAF не будет проверять запрос.

*   **Для AWS WAF** (согласно документации):
    *   Максимальный размер тела веб-запроса для Application Load Balancer и AWS AppSync: **8 КБ**
    *   Максимальный размер тела веб-запроса для CloudFront, API Gateway, Amazon Cognito, App Runner и Verified Access: **64 КБ**
*   **Из документации Azure:**
    *   Старые версии (Core Rule Set 3.1 или ниже) позволяют сообщения размером более 128 КБ при отключенной проверке тела запроса, но такие сообщения не проверяются на уязвимости. Для новых версий (Core Rule Set 3.2 или новее) то же самое можно сделать, отключив лимит максимального размера запроса. Когда запрос превышает лимит:
        *   Режим предотвращения: логирует и блокирует запрос.
        *   Режим обнаружения: проверяет до лимита, игнорирует остальное и логирует, если `Content-Length` превышает лимит.
*   **Из документации Akamai:**
    *   По умолчанию WAF проверяет только первые 8 КБ запроса. Лимит можно увеличить до 128 КБ, добавив Advanced Metadata.
*   **Из документации Cloudflare:**
    *   До 128 КБ.

**Пробелы в проверке статических ресурсов (например, GET-запросы к .js)**

Некоторые стеки CDN/WAF применяют слабую или нулевую проверку содержимого к GET-запросам для статических ресурсов (например, путей, заканчивающихся на `.js`), при этом все еще применяя глобальные правила, такие как ограничение скорости и репутация IP. В сочетании с автоматическим кэшированием статических расширений это может быть использовано для доставки или "посева" вредоносных вариантов, которые влияют на последующие HTML-ответы.

**Практические сценарии использования:**

1.  Отправить полезную нагрузку в ненадежных заголовках (например, User-Agent) в GET-запросе к пути `.js`, чтобы избежать проверки содержимого, а затем сразу запросить основной HTML, чтобы повлиять на кэшированный вариант.
2.  Использовать новый/чистый IP; как только IP будет помечен, изменения маршрутизации могут сделать технику ненадежной.
3.  В Burp Repeater используйте "Отправить группу параллельно" (в стиле одного пакета), чтобы отправить два запроса (`.js`, затем HTML) наперегонки через один и тот же путь фронтенда.

Это хорошо сочетается с отравлением кэша через отражение заголовков.

**Обфускация**

*   **IIS, ASP Classic:**
    `<%s%cr%u0131pt>` == `<script>`
*   **Обход черного списка путей - Tomcat:**
    `/path1/path2/` == `;/path1;foo/path2;bar/;`

**Совместимость Unicode (Unicode Compatibility)**

В зависимости от реализации нормализации Unicode (подробнее здесь), символы, имеющие совместимость в Unicode, могут обойти WAF и выполниться как предполагаемая полезная нагрузка. Совместимые символы можно найти здесь.

**Пример:**

```
# по алгоритму нормализации NFKD символы слева преобразуются в XSS-полезную нагрузку справа
＜img src⁼p onerror⁼＇prompt⁽1⁾＇﹥  -->  ＜img src=p onerror='prompt(1)'>
```

**Обход контекстных WAF с помощью кодировок**

Как упоминается в этом посте блога, чтобы обойти WAF, способные поддерживать контекст пользовательского ввода, мы можем использовать техники самого WAF для нормализации пользовательского ввода.

Например, в посте упоминается, что Akamai декодировал пользовательский ввод из URL 10 раз. Таким образом, что-то вроде `<input/%2525252525252525253e/onfocus` будет воспринято Akamai как `<input/>/onfocus`, что может показаться безопасным, так как тег закрыт. Однако, поскольку приложение не декодирует ввод из URL 10 раз, жертва увидит что-то вроде `<input/%25252525252525253e/onfocus`, что все еще является валидным для XSS-атаки.

Следовательно, это позволяет скрывать полезные нагрузки в закодированных компонентах, которые WAF декодирует и интерпретирует, в то время как жертва — нет.

Более того, это можно делать не только с URL-кодированными полезными нагрузками, но и с другими кодировками, такими как Unicode, hex, octal...

В посте предлагаются следующие финальные обходные пути:

*   **Akamai:** `akamai.com/?x=<x/%u003e/tabindex=1 autofocus/onfocus=x=self;x['ale'%2b'rt'](999)>`
*   **Imperva:** `imperva.com/?x=<x/\x3e/tabindex=1 style=transition:0.1s autofocus/onfocus="a=document;b=a.defaultView;b.ontransitionend=b['aler'%2b't'];style.opacity=0;Object.prototype.toString=x=>999">`
*   **AWS/Cloudfront:** `docs.aws.amazon.com/?x=<x/%26%23x3e;/tabindex=1 autofocus/onfocus=alert(999)>`
*   **Cloudflare:** `cloudflare.com/?x=<x tabindex=1 autofocus/onfocus="style.transition='0.1s';style.opacity=0;self.ontransitionend=alert;Object.prototype.toString=x=>999">`

Также упоминается, что в зависимости от того, как некоторые WAF понимают контекст пользовательского ввода, это можно использовать. Предложенный в блоге пример: Akamai разрешал помещать что угодно между `/*` и `*/` (вероятно, потому что это обычно используется как комментарии). Следовательно, SQL-инъекция типа `/*'or sleep(5)-- -*/` не будет обнаружена и будет работать, так как `/*` является начальной строкой инъекции, а `*/` закомментировано.

Такие проблемы с контекстом также могут быть использованы для эксплуатации других уязвимостей, помимо той, которую WAF ожидает обнаружить (например, это также может быть использовано для эксплуатации XSS).

**Пробелы в проверке первого оператора встроенного JavaScript (Inline JavaScript first-statement inspection gaps)**

Некоторые наборы правил для встроенного кода анализируют только первый оператор JavaScript внутри обработчика событий. Предваряя безвредное на вид выражение в круглых скобках, за которым следует точка с запятой (например, `onfocus="(history.length);payload"`), вредоносный код, помещенный после точки с запятой, обходит проверку, в то время как браузер все равно его выполняет. Сочетание этого с фокусировкой, вызванной фрагментом URL (например, добавление `#forgot_btn`, чтобы целевой элемент получал фокус при загрузке), позволяет осуществлять XSS без клика, который может немедленно вызвать `$.getScript` и загрузить фишинговые инструменты, такие как кейлоггеры.
